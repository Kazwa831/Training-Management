# CLAUDE.md

このファイルは、Claude Codeがこのリポジトリで作業する際のガイドラインです。

## 1. プロジェクト概要

**筋トレ管理アプリ(training-management)**

週複数回ジムに通う個人利用者が、自分の体調(疲労度・体重)と体型(身長・体脂肪率・目標)に応じて、その日の筋トレメニューをAIに提案してもらい、実施したセット数・回数・重量を記録・振り返るためのWebアプリ。スマホのブラウザからジムでもストレスなく使えることを重視する。

- 仕様の正本は **`docs/Training-Management-project-design-doc.md`**。実装で仕様に疑問が生じたら、まずこのドキュメントを確認する。
- ユーザーは本人1人のみの個人利用ツール(マルチユーザー対応は行わない)。

## 2. 開発目的

このプロジェクトは**就職活動用のポートフォリオ**として開発する。そのため以下を重視する。

- 動くものを作るだけでなく、コードの読みやすさ・設計の一貫性を示せること
- 実装の意図が第三者(採用担当者やレビュアー)にも伝わること
- 過度に複雑な実装より、シンプルで説明しやすい実装を優先する

## 3. 技術スタック

| 項目 | 選定 | 理由 |
|---|---|---|
| フロントエンド | Next.js (App Router) + TypeScript + Tailwind CSS v4 | Vercelとの親和性、Claude Codeでの実装実績の多さ |
| ホスティング | Vercel(無料Hobbyプラン) | GitHub連携で自動デプロイ、スマホからのアクセスに対応 |
| DB | Supabase(無料プラン、PostgreSQL) | 無料でクラウドDBを持て、Next.jsとの連携が容易 |
| DBアクセス経路 | Route Handler / Server Action経由のみ | クライアントにDB接続情報を渡さない。service role keyはサーバー側環境変数のみで使用 |
| AI | Anthropic Claude API(`@anthropic-ai/sdk`、サーバー側から呼び出し) | AIメニュー提案機能に使用。詳細は仕様書8章 |
| グラフ | Recharts | Reactとの親和性、必要十分な機能 |
| バリデーション | Zod | フォーム・API入力値の検証。Next.js公式ガイドでもServer Actionの検証手段として推奨されている |
| 認証 | 簡易パスコード方式(環境変数 + httpOnly Cookie) | 個人利用アプリのため本格的な会員認証は過剰。仕様書9章 |

### Next.js 16特有の注意点(重要)

このプロジェクトは Next.js 16 を使用している。学習データにある古いNext.jsの知識と異なる破壊的変更があるため、**実装前に必ず `node_modules/next/dist/docs/` 内の該当ガイドを確認すること**。

- **`middleware.ts` は廃止され `proxy.ts` に名称変更された**(関数名も `middleware()` ではなく `proxy()`)。このプロジェクトの認証ガード(全ページ/APIの保護)は `src/proxy.ts` として実装する。
- その他の規約変更が疑われる場合は `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md` を確認する。

## 5. 設計方針

- **保守性・可読性を最優先**にする。多少冗長でも、読んで意図が伝わるコードを優先する。
- **未経験者でも理解しやすいコード**を書く。複雑なテクニックや過度な抽象化を避け、素直な実装を選ぶ。
- 3行程度の似た処理を無理に共通化しない。実際に3箇所以上で重複し、かつ将来も同じ形で使う見込みがある場合のみ抽象化する。
- 今回のスコープにない機能(Phase2項目)は実装せず、必要な箇所にコメントで `// TODO(Phase2): ...` を残す程度に留める。
- 新しいライブラリを導入する際は、**導入理由を必ず説明**してから追加する(このファイルの技術スタック表を更新する)。

## 6. コーディングルール

- 言語: TypeScript。`any` は使わない(やむを得ない場合は理由をコメントで明記)。
- コメントは「なぜそうしているか」が非自明な場合のみ書く。コードを読めば分かることは書かない。
- 日本語コメント・日本語UIで統一する(仕様書・ユーザーが日本語のため)。
- スタイリングはTailwind CSSのユーティリティクラスを直接使う。UIトーンはプロトタイプを踏襲し、ダーク基調 + 種目タイプごとの色分け(下半身=amber、上半身プッシュ=rust、上半身プル=moss、有酸素+体幹=steel)とする。
- モバイル最優先のレスポンシブデザイン。タップ領域は44px以上を確保する(ジムでの片手操作を想定)。
- サーバー側の秘密情報(Anthropic APIキー、Supabase service role key、APP_PASSCODE)は`.env.local`にのみ記載し、クライアントバンドルに含めない(`NEXT_PUBLIC_`プレフィックスを付けない)。

## 7. コンポーネント設計ルール

- 画面(`page.tsx`)は「データ取得+レイアウト」に専念し、複雑なUIロジックは`src/components/<画面名>/`配下に切り出す。
- 汎用的に使えるUI部品(ボタン・カード・入力欄など)は`src/components/ui/`に置き、画面固有のロジックを持たせない。
- クライアントインタラクション(フォーム入力、タイマー等)が必要なコンポーネントにのみ`"use client"`を付ける。それ以外はデフォルトのServer Componentのままにする。
- 1コンポーネント1責務を意識し、propsの数が増えすぎる場合は分割を検討する。

## 8. API設計ルール

- DBアクセスは必ずRoute Handler(`src/app/api/**/route.ts`)またはServer Action経由で行い、クライアントから直接Supabaseへ接続しない。
- リクエストボディ・クエリパラメータはZodスキーマで検証してから処理する。
- レスポンスはJSON形式で統一し、エラー時は適切なHTTPステータスコード+エラーメッセージを返す。
- `/api/suggest`はAnthropic Messages APIをサーバー側から呼び出す。JSON出力のパースに失敗した場合・API呼び出しに失敗した場合は、ルールベースの標準メニュー生成にフォールバックする(仕様書8.3節)。
- 全ページ・全APIは`src/proxy.ts`(Next.js 16の`proxy`規約)でパスコード認証Cookieの有無をチェックし、未認証時は`/login`にリダイレクト(APIは401)する。

## 9. DB設計ルール

- テーブル定義は仕様書6章のSQLを正とする。マイグレーションが必要な場合は変更内容を`docs/progress.md`に記録する。
- 個人利用の単一ユーザーアプリのため`user_id`は持たない(将来マルチユーザー化する場合に追加する設計)。
- 種目カタログはDBテーブル化せず、`src/lib/exercises.ts`にコード内定数として持つ。
- 日付(`date`)をユニークキーとするテーブル(`body_metrics`, `workout_logs`)は「1日1件」の前提を崩さないよう、upsert(`date`をキーにした更新)で記録する。

## 10. 実装順序

1. プロジェクトの基盤構築(Next.js/Tailwind/依存関係セットアップ、ディレクトリ構成) ← 完了
2. Supabaseプロジェクト作成・テーブル作成・接続確認
3. 認証(パスコードログイン + `proxy.ts`によるページ/API保護)
4. プロフィール画面(身長・体脂肪率・目標の登録、BMI計算)
5. 今日画面:体重・疲労度の記録(手動入力)
6. 種目カタログ定数の整備 + 手動メニュー選択・セット記録・保存
7. AIメニュー提案(`/api/suggest`)+ ルールベースフォールバック
8. セット間インターバルタイマー
9. 履歴画面(一覧・詳細表示)
10. グラフ画面(体重・体脂肪率・疲労度推移、種目別重量推移)
11. 全体の動作確認・レスポンシブ調整・Vercelデプロイ
