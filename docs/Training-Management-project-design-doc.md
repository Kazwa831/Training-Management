# 筋トレ管理アプリ 設計書 (v1)

## 0. この文書について

前段階で作った claude.ai 上のプロトタイプ(単一HTMLファイル、ブラウザ内ストレージ+AI提案機能)をベースに、Claude Codeへ実装を依頼するための設計書。プロトタイプで検証済みのUI/ロジックの方向性は踏襲しつつ、スマホからジムでも使える形にホスティングし、体型情報を考慮した提案ができるように拡張する。

---

## 1. 目的・ゴール

- 週複数回ジムに通う本人が、自分の体調(疲労度・体重)と体型(身長・体脂肪率・目標)に応じて、その日の筋トレメニューをAIに提案してもらう
- 実施したセット数・回数・重量を記録し、履歴として振り返れるようにする
- スマホのブラウザから、ジムでもストレスなく入力・閲覧できる

## 2. 想定ユーザー・利用シーン

- ユーザーは本人1人のみ(個人利用ツール)
- 主な利用タイミングはジムでの運動前後(スマホ操作)、たまにPCから振り返り閲覧
- 個人の記録を扱うため、第三者からのアクセスは防ぎたい(簡易ログインで対応、詳細は9章)

## 3. スコープ

| 区分 | 内容 |
|---|---|
| v1 (今回Claude Codeに依頼する範囲) | 4章の「MVP」マーク付き機能すべて |
| Phase2以降 | 4章の「Phase2」マーク付き機能。v1リリース後、使用感を見ながら追加 |

## 4. 機能一覧

### 4.1 コア機能(プロトタイプから踏襲・MVP)

- 日次記録:体重・疲労度(5段階)・種目ごとのセット数/回数/重量・有酸素の実施時間・自由メモ
- AIによる当日メニュー提案(種目・セット数・回数・重量目安・提案理由コメント)
- 手動で種目タイプ(下半身/上半身プッシュ/上半身プル/有酸素+体幹)を選んで記録するモードとの併用
- 履歴一覧・詳細表示(日付ごとに展開して確認)
- グラフ表示(体重・疲労度の推移、種目別の重量推移)

### 4.2 新規:体型プロファイル対応(MVP)

- プロフィール画面で以下を設定・更新
  - 身長(cm)
  - 体脂肪率(%) ※体重と同様に日付ごとの推移として記録し、グラフ化対象にする
  - 目標:増量 / 減量 / 維持 のいずれか
- BMIを身長・体重から自動算出し、プロフィール画面と履歴に軽く表示(参考値であることを明記)
- AIへの提案リクエストに上記を含め、目標に応じてメニューの重量・回数・有酸素の配分を変える(詳細は8章)

### 4.3 追加で検討してほしい機能

もともとの依頼になかったが、実際にジムで使う前提だと効果が大きいと思われる順に整理した。MVPに入れる/Phase2に回すは相談して決めたい。

| 機能 | 内容 | 提案区分 | 理由 |
|---|---|---|---|
| セット間インターバルタイマー | 種目カード内から休憩時間を計測、終了時にバイブ/音で通知 | **MVP推奨** | ジムでの実利用頻度が一番高い機能。スマホのタイマーアプリと行き来する手間をなくせる |
| 簡易ログイン(パスコード保護) | 環境変数の合言葉でログイン、以降はCookieで保持 | **MVP推奨** | 無料ホスティングで公開URLになるため、最低限のアクセス制御は必須に近い |
| 自己ベスト(PR)自動検出 | 種目ごとに過去最大重量を更新したら「自己ベスト更新」を表示 | Phase2 | モチベーション維持に効果的だが必須ではない |
| ストリーク(連続日数)表示 | 継続記録日数をトップに表示 | Phase2 | 同上 |
| 週次/月次サマリー | 総トレーニング回数・総有酸素時間・種目別ボリュームの振り返り | Phase2 | データが溜まってから効果が出る機能 |
| カレンダービュー | 月間でどの日に何を実施したか一覧表示 | Phase2 | 履歴一覧で当面代替可能 |
| AI提案への👍👎フィードバック | 提案が良かったかを記録し、次回提案のプロンプトに反映 | Phase2 | 提案精度向上には有効だが、まずは体型考慮版の精度を見てから検討 |
| PWA化・オフラインキャッシュ | ホーム画面追加、直近メニューのオフライン閲覧・記録 | Phase2 | ジムのWi-Fi不安定対策として有用 |
| データエクスポート(CSV/JSON) | 記録のバックアップ・他サービス連携用 | Phase2 | 個人利用でもバックアップの安心感はある |
| 睡眠時間・食欲などの簡易コンディション項目 | 任意入力、AIの提案精度向上に使う | Phase2 | 入力の手間が増えるため、まずは体重・疲労度・体型だけで様子見 |

## 6. データモデル(Supabase / PostgreSQL想定)

個人利用の単一ユーザーアプリのため、テーブルはシンプルにユーザーIDを持たず固定1系統として設計する(将来的にマルチユーザー化する場合は各テーブルに user_id を足す想定)。

```sql
-- プロフィール(1行のみ運用)
create table profile (
  id           smallint primary key default 1,
  height_cm    numeric,
  goal         text check (goal in ('bulk','cut','maintain')),
  updated_at   timestamptz default now()
);

-- 体重・体脂肪率・疲労度(日付ごと。ワークアウトをしない日も記録可)
create table body_metrics (
  id             uuid primary key default gen_random_uuid(),
  date           date unique not null,
  weight_kg      numeric not null,
  body_fat_pct   numeric,
  fatigue_level  smallint check (fatigue_level between 1 and 5),
  created_at     timestamptz default now()
);

-- ワークアウト記録(1日1件)
create table workout_logs (
  id                    uuid primary key default gen_random_uuid(),
  date                  date unique not null,
  day_type              text check (day_type in ('legs','push','pull','cardio_core')),
  ai_comment            text,
  cardio_exercise_id    text,
  cardio_target_minutes numeric,
  cardio_actual_minutes numeric,
  note                  text,
  feedback              text check (feedback in ('good','bad')), -- Phase2で使用
  created_at            timestamptz default now()
);

-- ワークアウト内の種目ごとのセット記録
create table workout_sets (
  id                 uuid primary key default gen_random_uuid(),
  workout_log_id     uuid references workout_logs(id) on delete cascade,
  exercise_id        text not null,       -- カタログのid(下記7章参照)
  set_index          smallint not null,
  target_reps        smallint,
  target_weight_hint numeric,
  actual_reps        smallint,
  actual_weight      numeric,
  note               text
);
```

種目カタログ(下半身6・上半身11・体幹3・有酸素4)はDB化せず、プロトタイプと同様にコード内定数として持つ(ジムの設備が変わった時だけコード修正すればよいため)。

## 7. アーキテクチャ・技術スタック

| 項目 | 選定 | 理由 |
|---|---|---|
| フロントエンド | Next.js (App Router) + TypeScript + Tailwind CSS | Vercelとの親和性が高く、Claude Codeでの実装実績も多い |
| ホスティング | Vercel(無料Hobbyプラン) | 「スマホでジムでも使いたい」に対応、GitHub連携で自動デプロイ |
| DB | Supabase(無料プラン、Postgres) | 無料でクラウドDBが持てる、Next.jsとの連携が容易 |
| DBアクセス経路 | Next.jsのRoute Handler/Server Action経由のみ | クライアントに直接DB接続情報を渡さず、Supabaseの service role key はサーバー側の環境変数のみで使用 |
| AI | Anthropic Claude API(サーバー側から呼び出し) | 8章参照 |
| グラフ | Recharts | Reactとの親和性、必要十分な機能 |
| 認証 | 簡易パスコード方式 | 9章参照 |

## 8. AI連携仕様

### 8.1 呼び出し方法

- サーバー側の `/api/suggest` などのRoute HandlerからAnthropic Messages APIを呼び出す(APIキーはクライアントに露出させない)
- モデルは `claude-sonnet-5` を推奨(2026年7月時点の最新Sonnet系、コーディング/指示追従のバランスが良い。1日1回程度の呼び出し頻度なのでコストはほぼ気にしなくてよい規模)
- 出力はプロトタイプ同様、JSON形式のみを指定し、パース失敗時はルールベースの標準メニューにフォールバックする

### 8.2 プロンプトに追加する体型・目標情報

これまでの「体重・疲労度・直近の記録」に加えて、以下をsystemプロンプト/userプロンプトに追加する。

```
- 身長: {height_cm}cm
- 直近の体脂肪率: {body_fat_pct}%（未記録の場合は省略）
- 目標: {goal}（増量 / 減量 / 維持）
```

目標別のメニュー調整方針をsystemプロンプトに明記する:

| 目標 | 調整方針 |
|---|---|
| 増量 | 高重量・中〜低回数を優先。有酸素は最小限(回復目的の軽い有酸素のみ)に留める |
| 減量 | 回数をやや増やし、有酸素の時間配分を増やす。ただし疲労度が高い日は無理にボリュームを増やさない |
| 維持 | 現状のバランス型メニュー(プロトタイプのデフォルト方針)を継続 |

体脂肪率の推移が目標と逆行している場合(例:減量目標なのに体脂肪率が上昇傾向)は、コメント欄で押し付けがましくない範囲で軽く触れるよう指示する。

### 8.3 フォールバック

AI呼び出し失敗時・JSONパース失敗時は、プロトタイプのルールベースロジック(ローテーション+疲労度による重量調整)をそのまま踏襲する。体型・目標を使った調整はAI提案時のみとし、ルールベース版は簡易調整に留めてよい。

## 9. 認証・セキュリティ

- Vercelにデプロイすると誰でもアクセスできるURLになるため、最低限のアクセス制御を入れる
- 方式:環境変数 `APP_PASSCODE` と照合するログイン画面 → 一致したらhttpOnly Cookieを発行(有効期限は長め、例30日)し、以降 `middleware.ts` で全ページ/APIを保護する
- 本格的な会員登録機能(メール認証など)は個人利用には過剰なため導入しない
- Anthropic APIキー・Supabaseキーはすべてサーバー側の環境変数として管理し、クライアントバンドルに含めない

---

## 10. デプロイ構成

1. GitHubリポジトリを作成し、Claude Codeで実装したコードをpush
2. Vercelでそのリポジトリと連携し、pushごとに自動デプロイ
3. Supabaseで無料プロジェクトを作成し、6章のSQLでテーブルを作成
4. Vercelの環境変数に以下を登録
   - `ANTHROPIC_API_KEY`
   - `SUPABASE_URL`
   - `SUPABASE_SERVICE_ROLE_KEY`
   - `APP_PASSCODE`
5. スマホはURLをブラウザで開き、ホーム画面に追加してアプリのように利用(アイコン・PWA化はPhase2)

---

## 11. 非機能要件

- モバイル最優先のレスポンシブデザイン。ジムでの片手操作を想定し、タップ領域は44px以上を確保
- AI提案のレスポンス待ちは数秒程度を許容し、ローディング表示を出す
- 将来のPhase2(オフライン対応・エクスポート)を見据え、データはすべてDB経由で一元管理し、フロントに秘密情報を持たせない構成にしておく

## 13. Claude Codeへの申し送り事項

- 参考実装として、claude.ai上で作ったプロトタイプ(単一HTMLファイル、ブラウザ内ストレージ+AI提案ロジック)を別途共有する。UIのトーン(ダーク基調、種目タイプごとの色分け:下半身=amber、上半身プッシュ=rust、上半身プル=moss、有酸素+体幹=steel)とAIプロンプトの構造はこのプロトタイプを踏襲しつつ、Next.js + Supabase構成に置き換えて実装してほしい
- 種目カタログ(下半身6/上半身11/体幹3/有酸素4)は固定値としてコード内に定義し、DBテーブル化はしない
- v1のスコープは4章で「MVP」とした機能のみ。Phase2項目は今回のスコープ外として、コメントでTODOを残す程度でよい
